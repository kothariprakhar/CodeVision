1Model Overview

What This Model Does

The CodeVision AI Analysis Engine is a multi-pass LLM-powered pipeline that transforms raw GitHub repositories into structured, business-friendly architecture intelligence. It combines deterministic code analysis (AST parsing, dependency graphs, pattern detection) with progressive AI reasoning across 4 analysis passes to produce architecture diagrams, business capability maps, risk assessments, and executive narratives.

Unlike traditional static analysis tools that only detect code-level patterns, this system bridges the gap between technical implementation and business understanding, making codebases accessible to founders, product managers, and investors who need to evaluate software architecture without reading code.

🧠
Core LLM — Claude Sonnet

Anthropic Claude Sonnet as reasoning backbone. Used for code understanding, architecture inference, business context generation, and interactive Q&A. Not fine-tuned — uses carefully engineered system prompts with structured JSON output schemas.

🌳
AST Parser — Tree-sitter

Tree-sitter and ast-grep for deterministic structural analysis. Parses source code into abstract syntax trees to extract classes, functions, imports, and exports. Builds dependency graphs and detects architecture patterns (MVC, Clean Architecture, microservices).

🔍
Business Mapper

Keyword-based capability mapping engine that translates technical architecture nodes into 8 business capabilities across 3 strategic domains. Uses confidence scoring (0.45–0.95) and generates deterministic user journey graphs as fallback.

2Intended Use

The model serves as the core intelligence engine within CodeVision, performing automated code architecture analysis that would traditionally require a senior engineer spending hours or days reviewing a codebase.

Non-Technical Founders
Understand what their engineering team has built, evaluate technical decisions, and communicate architecture to investors.
"What does my codebase actually do and how risky is the architecture?"
Product Managers
Map technical systems to business capabilities, identify user journey gaps, and assess feature delivery risk.
"Which business capabilities are well-supported vs. underdeveloped?"
Technical Due Diligence
Rapidly assess technology stack, architecture patterns, risk factors, and technical debt before investment or acquisition.
"What are the top risks in this codebase and what would remediation cost?"
Engineering Teams
Onboard onto new codebases, understand dependency structures, track architecture evolution across versions.
"How does data flow from API to database? What changed between v1 and v2?"
Out of Scope
— Automated code generation or modification
— Security vulnerability scanning (SAST/DAST)
— Runtime performance profiling or load testing
— Compliance auditing (SOC2, HIPAA, GDPR)
— Replacing human code review for merge decisions

Known Data Limitations
  1. File selection capped at 400 — very large monorepos may have important files excluded from analysis
  2. Tree-sitter AST parsing only covers 5 languages natively; others fall back to regex with lower accuracy
  3. Binary files, minified code, and auto-generated code may be included, potentially skewing results
  4. Private repositories require GitHub tokens; token permissions may limit metadata access
  5. Requirement documents are optional — without them, gap analysis is not possible

Evaluation Methodology

We plan to evaluate against a benchmark of 20 open-source repositories of varying size and complexity (small utilities, medium web apps, large frameworks). For each repo, a senior engineer produces a ground-truth architecture map, and we measure node/edge overlap, technology detection F1, and risk recall. Business narrative quality uses 5-point Likert ratings from non-technical reviewers. The benchmark suite is under active development.

5Performance & Limitations
Where It Excels
  • Standard Web Applications
    React/Next.js, Django/FastAPI, Express/Node.js apps with clear frontend-backend-database tiers.
  • Common Architecture Patterns
    MVC, MVVM, Clean Architecture, REST APIs, and monolithic structures are reliably detected.
  • Technology Stack Identification
    92% precision on the 44-technology fingerprint library. Framework and infrastructure detection is highly accurate.
  • Business Translation
    Strong at converting technical structures into plain-English narratives for non-technical stakeholders.
  • Small-to-Medium Repos
    Best results on 50–500 source files. The 400-file window covers most of the codebase.
Where It Struggles
  • Very Large Monorepos
    Repos with 1000+ files may have critical modules excluded, leading to incomplete architecture maps.
  • Uncommon Languages
    Languages without Tree-sitter support (Elixir, Haskell, Scala) rely on regex parsing, missing nested structures.
  • Microservice Architectures
    Each microservice in a separate repo is analyzed independently — cross-service patterns are not visible.
  • Heavy Metaprogramming
    Dynamic imports, reflection, and runtime module loading are invisible to static analysis.
  • AI Hallucination Risk
    LLM may infer connections that don't exist, especially for unfamiliar frameworks or domain-specific patterns.

Known Failure Modes

High
Rate Limit Exhaustion
Multiple concurrent analyses can exhaust Claude API rate limits, causing passes 2–4 to fail.
Mitigation: Exponential backoff (3 retries), single-pass fallback, and job queuing.
Medium
Token Overflow
Very complex modules with deeply nested code can exceed the 80K token-per-pass limit, requiring truncation that loses context.
Mitigation: File prioritization and chunking limits total context; fallback to single-pass mode.
Low
Circular Dependencies
Heavily circular import graphs can confuse the topological sorting for layout, producing tangled diagrams.
Mitigation: Force-directed graph layout with cycle-breaking heuristics.
Low
Stale Cached Results
Analysis results are cached for 24h. Rapid code changes may show outdated architecture.
Mitigation: Users can trigger re-analysis; version diff view shows changes between analyses.
Low
Empty / Stub Repositories
Repos with very few files or only boilerplate produce minimal or low-confidence analysis.
Mitigation: Quality report flags low-evidence analyses with explicit confidence warnings.
Bias Considerations

The system may exhibit bias toward popular frameworks well-represented in Claude's training data. Niche technologies, proprietary frameworks, or domain-specific patterns (embedded systems, HPC, game engines) may be under-analyzed or misclassified. The 44-technology fingerprint library reflects a web-development-centric view. Business capability mapping uses keyword heuristics that favor SaaS business models.

6Improvement Path
Implemented Improvements
Multi-Pass Pipeline (from Single-Pass)
Splitting into 4 focused passes improved architecture accuracy by reducing cognitive load per LLM call. Each pass works within the 80K token limit while maintaining coherence through context chaining.
Deterministic + AI Hybrid Approach
Adding Tree-sitter AST parsing and static heuristics as a foundation layer reduced AI hallucination. The LLM now refines deterministic signals rather than generating architecture from scratch.
Fallback Strategy
When the multi-pass pipeline fails (rate limits, API errors), automatic fallback to single-pass analysis ensures users always get results. Quality degrades gracefully rather than failing completely.
Quality Report & Confidence Scoring
Each analysis now includes coverage score, evidence density, and low-confidence flags. Users can see where the system is uncertain rather than trusting all outputs equally.
